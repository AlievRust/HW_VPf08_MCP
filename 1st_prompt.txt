Идея: создать полный рабочий учебный MCP-проект: notes-mcp, состоящий из двух компонентов:

1. MCP-сервер.
Функции сервера:
- использовать локальную SQLite3 базу данных
- создать таблицу notes с колонками:
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  title TEXT,
  content TEXT,
  tags TEXT,
  created_at TEXT,
  updated_at TEXT
- при первом запуске автоматически заполнить таблицу 20 тестовыми заметками
- реализовать MCP-инструменты:
  list_notes() → возвращает список заметок
  get_note(id: int) → возвращает одну заметку
  create_note(title: string, content: string, tags: string) → создаёт заметку
  search_notes(query: string) → ищет заметки по title/content/tags
  update_note(id: int, title: string, content: string, tags: string) → обновляет заметку
  delete_note(id: int) → удаляет заметку

Требования:
- сервер должен быть MCP-совместимым
- инструменты должны экспортироваться в формате MCP JSON schema
- запускаться командой: python server.py
- база notes.db создаётся автоматически при запуске
- eval запрещён
- код должен быть простым, учебным и хорошо прокомментированным

Структура:
notes_mcp/
├── server.py
├── db.py
├── tools.py
├── notes.db
└── requirements.txt
2. CLI CLIENT для работы с notes-mcp.

Клиент должен:

* работать в терминале как интерактивный чат

* запускаться командой:

  python cli.py

* подключаться к MCP серверу notes-mcp

* использовать модель OpenAI gpt-5.4 или другую модель, указанную в .env

* при сообщении пользователя:
  → CLI-клиент отправляет текст пользователя в LLM
  → LLM анализирует запрос пользователя
  → LLM решает, нужно ли вызвать MCP tool
  → если нужно, CLI-клиент вызывает соответствующий MCP tool
  → результат tool-вызова передаётся обратно в LLM
  → CLI-клиент печатает финальный ответ человеку в терминал

Пользователь может писать обычным языком, например:

* "покажи все заметки"
* "найди заметки про vps"
* "создай заметку про настройку сервера"
* "создай заметку с названием MCP идеи и тегами mcp, agent, python"
* "покажи заметку 3"
* "обнови заметку 5"
* "удали заметку 7"
* "найди всё, где упоминается SQLite"
* "какие заметки есть по теме автономного агента?"

CLI-клиент должен:

* отвечать красиво и структурировано
* не показывать пользователю сырой JSON, если не включён debug-режим
* если запрос непонятен — задавать уточняющий вопрос
* поддерживать историю текущего диалога в памяти процесса
* поддерживать команды:

  /exit — выйти из программы
  /help — показать список возможностей
  /tools — показать доступные MCP tools
  /debug on — включить вывод отладочной информации
  /debug off — выключить вывод отладочной информации

Структура CLI-клиента:

cli_client/
├── cli.py
├── config.py
├── mcp_client.py
├── llm_client.py
└── requirements.txt

Файлы:

config.py:

* загружает настройки из .env
* хранит OPENAI_API_KEY
* хранит OPENAI_MODEL
* хранит MCP_SERVER_URL или параметры подключения к MCP серверу
* валидирует наличие обязательных переменных окружения

mcp_client.py:

* подключается к notes-mcp серверу
* получает список доступных MCP tools
* вызывает MCP tools по имени
* передаёт аргументы в MCP tools
* обрабатывает ошибки MCP сервера
* возвращает результат tool-вызова в удобном формате

llm_client.py:

* отправляет сообщения пользователя в OpenAI API
* передаёт модели schemas доступных MCP tools
* обрабатывает tool calls от модели
* определяет, какой MCP tool нужно вызвать
* передаёт результат MCP tool обратно в модель
* возвращает финальный человекочитаемый ответ

cli.py:

* запускает интерактивный цикл ввода пользователя
* принимает команды пользователя
* печатает ответы модели
* обрабатывает команды /exit, /help, /tools, /debug on, /debug off
* хранит историю текущего диалога в памяти процесса
* аккуратно обрабатывает KeyboardInterrupt и EOFError

3. ДОП. УСЛОВИЯ

Использовать .env файл для хранения ключей и настроек.

Добавить .env.example без реальных ключей:

OPENAI_API_KEY=your_openai_api_key_here
OPENAI_MODEL=gpt-5.4
MCP_SERVER_URL=http://localhost:8000
OPENAI_BASE_URL=https://api.openai.com/v1

Проверить и добавить при необходимости в .gitignore:
.env
*.db
**pycache**/
.venv/
venv/
.pytest_cache/

Добавить README.md с инструкциями по запуску проекта:

1. создать виртуальное окружение
2. установить зависимости MCP сервера
3. установить зависимости CLI-клиента
4. создать .env файл на основе .env.example
5. запустить MCP сервер
6. запустить CLI-клиент
7. проверить работу команд:

   * "покажи все заметки"
   * "создай заметку про MCP"
   * "найди заметки про MCP"

Код должен быть полностью готовым к запуску.

Требования к безопасности и качеству:

* секреты нельзя хардкодить в коде
* пользовательский ввод нужно валидировать
* ошибки должны обрабатываться понятными сообщениями
* при недоступности MCP сервера CLI-клиент должен вывести понятную ошибку
* при отсутствии OPENAI_API_KEY CLI-клиент должен вывести понятную ошибку
* если модель вернула некорректный tool call, CLI-клиент должен обработать это без падения
* если MCP tool вернул ошибку, CLI-клиент должен передать её модели и сформировать понятный ответ пользователю

DEBUG-режим:

Если включён debug-режим, CLI-клиент должен выводить:

* исходный запрос пользователя
* какой MCP tool решила вызвать модель
* какие аргументы были переданы
* какой результат вернул MCP сервер
* финальный ответ модели

Если debug-режим выключен, пользователь должен видеть только обычный красивый ответ.

Важно:

* Не реализовывать MCP tools как произвольные REST endpoints вида /create_note или /search_notes.
* Использовать MCP-совместимый способ объявления и вызова tools.
* Если для учебного упрощения используется HTTP transport, явно отделить MCP layer от обычного HTTP API.
* MCP сервер должен предоставлять tools, а CLI-клиент должен получать их список и передавать schemas модели.
* CLI-клиент не должен знать бизнес-логику заметок напрямую. Он должен обращаться к notes-mcp через MCP tools.
