# Технический обзор

## Архитектура

Проект состоит из двух независимых частей:

1. `mcp_server/` - локальный MCP-сервер заметок.
2. `cli_client/` - CLI-клиент, который общается с OpenAI и вызывает MCP tools.

## Поток данных

1. Пользователь вводит текст в CLI.
2. CLI отправляет запрос в OpenAI Responses API.
3. Модель либо отвечает текстом, либо возвращает `function_call`.
4. CLI вызывает соответствующий MCP tool через `MCPClient`.
5. Результат tool call возвращается в модель как `function_call_output`.
6. Модель формирует финальный человекочитаемый ответ.

## MCP-сервер

Сервер использует `FastMCP` и transport `streamable-http`.

Хранилище данных:
- SQLite-файл `mcp_server/notes.db`
- таблица `notes`
- автосоздание схемы при первом запуске
- автоматическое заполнение 20 тестовыми заметками, если база пустая

## CLI-клиент

CLI:
- загружает настройки из `.env`
- валидирует `OPENAI_API_KEY` и `OPENAI_MODEL`
- подключается к MCP-серверу по `MCP_SERVER_URL`
- получает список MCP tools и преобразует их в OpenAI function tools
- обрабатывает команды `/help`, `/tools`, `/debug on`, `/debug off`, `/exit`

## Формат tool calls

MCP tools возвращают обычные JSON-совместимые структуры.
CLI преобразует результат tool call в JSON-строку и передает ее обратно в Responses API.

## Почему именно так

Этот вариант оставляет логику заметок на стороне MCP-сервера, а CLI делает только orchestration:
- не знает бизнес-логику заметок;
- не ходит напрямую в SQLite;
- работает как тонкая оболочка над MCP и OpenAI.
